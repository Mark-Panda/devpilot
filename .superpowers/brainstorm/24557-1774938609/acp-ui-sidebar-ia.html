ACP（方案 B）：侧栏信息架构 & 交互组织

选一个“侧栏分组方式”。主区保持「流式输出 + 控制条」不变。

B1

状态优先（推荐）

把“我现在在第几轮/是否卡住/能做什么”放最上面；交互与诊断往下沉。

侧栏结构预览
状态
round / stopReason / 用时 / 超时倒计时
交互
ask_question / after_round（最新一条置顶）
交互历史（可展开）
诊断
stderr tail（默认折叠，异常时自动展开）
任务信息
规则 / 节点 / cwd / sessionId（复制）

适用

  • 长会话/多轮最顺手
  • “卡住了怎么处理”最直观

注意

  • 交互区需要很克制（避免变成第二个主区）
B2

交互优先

把“需要你做决定的事情”放最上面（问答/续聊/计划确认）；状态放第二层。

侧栏结构预览
交互（置顶）
当前 ask_question（选项）/ after_round（继续/结束）
状态
round / stopReason / 用时
诊断
stderr tail（折叠）
任务信息
规则 / 节点 / cwd（复制）

适用

  • 频繁“人机对话续聊”的工作流
  • 希望像“收件箱”一样处理事件

注意

  • 没交互时顶部会显空，需要良好空态
B3

诊断优先（偏工程排障）

把 stderr tail / 丢弃行统计 / stopReason 解释放上面，适合经常排查失败原因的团队。

侧栏结构预览
诊断（置顶）
stderr tail · 解析丢弃原因 · 复制诊断包
状态
round / stopReason / 用时
交互
ask_question / after_round（历史）
任务信息
规则 / 节点 / cwd（复制）

适用

  • 失败率较高/经常排障
  • 希望把“为什么失败”更快暴露

注意

  • 日常成功路径会显得“太工程化”
主区（固定基线，不用选）
流式输出为主 + 顶部控制条（暂停滚动 / 只看尾部 / 搜索 / 复制范围）
