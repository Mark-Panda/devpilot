# Proposal: 迁移到 Flowgram.ai 自由布局编辑器

## 概述

将 DevPilot RuleGo 规则链编辑器从 Blockly/Scratch-blocks 架构完全重写为 Flowgram.ai 自由布局编辑器，以获得更现代的视觉效果、更灵活的节点布局、更优秀的用户体验。

## 背景与动机

### 当前痛点

1. **视觉效果受限**
   - Blockly SVG 块渲染，样式定制能力有限
   - 固定的拼图式布局，视觉呆板
   - 无法实现现代 UI 设计（圆角、投影、渐变等）

2. **交互体验不佳**
   - 块的拼接式连接，限制了布局自由度
   - 配置面板以模态框形式弹出，割裂工作流
   - Loop 等容器节点以凹槽形式呈现，不够直观

3. **扩展性差**
   - Blockly 的块定义需要处理底层 SVG 渲染
   - 动态节点（如 Switch、GroupAction）依赖 mutation API，复杂易错
   - 与 React 生态集成困难

4. **代码维护成本高**
   - RuleGoScratchEditorPage.tsx 已达 7131 行
   - DSL 转换逻辑与 Blockly API 紧耦合
   - 33 个块定义分散在多个文件，缺乏统一范式

### 目标收益

1. **现代化的视觉体验**
   - 使用 Flowgram 的白色卡片式节点设计
   - 圆角、投影、渐变等现代 UI 元素
   - Loop 容器节点以真实的画布容器呈现

2. **更灵活的布局**
   - 节点可任意拖放到画布任何位置
   - 通过连线而非拼接表达流程关系
   - 支持节点分组、对齐辅助线等高级功能

3. **更好的开发体验**
   - React 组件化的节点定义，清晰直观
   - TypeScript 全链路类型安全
   - 插件化架构，功能解耦易扩展

4. **保持后端兼容**
   - RuleGo DSL 格式完全不变
   - 现有所有规则链无需迁移即可加载
   - 后端引擎无需任何修改

## 范围界定

### In Scope

- 完全重写前端规则链编辑器 UI
- 将 33 个 Blockly 块定义转换为 FlowNodeRegistry
- 实现 DSL 双向转换适配层
- 迁移所有现有功能（保存、导入、导出、Agent 规划等）
- 应用 Flowgram demo-free-layout 的视觉样式
- 重点优化 Loop 容器节点的视觉呈现

### Out of Scope

- 后端 RuleGo 引擎改动（保持 100% 兼容）
- 现有规则链 DSL 格式变更
- 数据库 schema 变更
- 执行引擎逻辑变更
- 其他页面（规则列表、执行日志等）改动

### Dependencies

- Flowgram.ai 开源库（MIT License）
- Semi Design UI 库
- Styled Components
- Inversify DI 框架

## 技术方案概览

### 架构对比

```
┌─────────────────────────────────────────────────────────────────┐
│                      架构演进                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  旧：Blockly SVG 块 → BlockTypeDef → DSL                        │
│                                                                 │
│  新：React 节点组件 → FlowNodeRegistry → DSL                    │
│                                                                 │
│  共同点：DSL 格式完全一致，后端无感知                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 核心技术栈

**新增依赖**：
- `@flowgram.ai/free-layout-editor` - 核心编辑器引擎
- `@flowgram.ai/*-plugin` - 插件生态（连线、对齐、小地图、容器等）
- `@douyinfe/semi-ui` - Semi Design 组件库
- `styled-components` - CSS-in-JS 样式方案
- `inversify` + `reflect-metadata` - 依赖注入

**移除依赖**：
- `scratch-blocks` - Blockly 编辑器

**保持不变**：
- React 18
- TypeScript
- Vite
- 与后端的所有 API 调用

### 关键设计决策

1. **保持 DSL 格式不变**
   - 适配层负责 Flowgram 内部数据 ↔ RuleGo DSL 转换
   - 后端完全无感知，现有规则链零迁移成本

2. **渐进式切换**
   - Feature Flag 控制新旧编辑器
   - 两套编辑器短期并存（2-4周）
   - 充分测试后再删除旧代码

3. **优先实现高频节点**
   - 先实现 HTTP 触发、LLM、循环、分支等核心节点
   - 低频节点（RPA、追踪类）后续迭代

## 风险与缓解

### 高风险

**风险 1: DSL 转换出错导致规则链损坏**
- 缓解：完整的单元测试 + 集成测试
- 缓解：所有生产规则的回归测试
- 缓解：保存前 DSL diff 对比确认

**风险 2: 容器节点（Loop/GroupAction）实现复杂度高**
- 缓解：优先攻克 Loop 节点，建立范式
- 缓解：深度研究 Flowgram 的 container-plugin 源码
- 缓解：预留充足开发和测试时间

**风险 3: 开发周期长（8周），期间业务需求阻塞**
- 缓解：Feature Flag 方案，新功能先在旧编辑器实现
- 缓解：分阶段交付，核心节点先上线
- 缓解：双编辑器并存期支持快速回滚

### 中风险

**风险 4: Flowgram 库的学习曲线**
- 缓解：充分研究 demo-free-layout 示例
- 缓解：建立标准化的节点转换模板
- 缓解：编写详细的开发文档

**风险 5: 性能问题（大规则链）**
- 缓解：使用 Flowgram 内置的虚拟化渲染
- 缓解：懒加载节点表单和配置
- 缓解：性能基准测试和监控

### 低风险

**风险 6: 第三方依赖更新破坏兼容**
- 缓解：锁定 Flowgram 依赖版本
- 缓解：定期关注上游更新和 breaking changes

## 备选方案

### 备选 1: CSS 深度定制 Blockly（不采纳）

**方案**：在现有 Blockly 基础上通过 CSS 覆盖样式

**优点**：
- 工作量小（1-2周）
- 无需重写代码
- 风险低

**缺点**：
- 效果有限，无法实现真正的现代 UI
- 块的拼图形状无法改变
- Loop 容器仍是凹槽形式
- 技术债务持续积累

### 备选 2: 混合架构（不采纳）

**方案**：部分节点用 Flowgram，其他保留 Blockly

**优点**：
- 可以先实现 Loop 等重点节点
- 渐进式迁移

**缺点**：
- 架构复杂，维护成本高
- 两套系统并存，代码重复
- 事件处理和状态同步困难

## 决策

**推荐方案**：完全重写（本 Proposal）

**理由**：
1. 长期来看技术债务最低
2. 用户体验提升最显著
3. 代码质量和可维护性最好
4. 符合现代前端工程化最佳实践
5. 虽然初期投入大，但后续扩展成本低
